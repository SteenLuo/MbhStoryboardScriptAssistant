# 综合参考判断：AI漫剧剧本分镜技能系统

本文档结合三个来源，形成当前项目自己的判断：

- `0WcAssistant` 既有讨论
- `garrytan/gstack`
- `garrytan/gbrain`

结论先行：本项目不做“大而全个人助理”，也不做完整知识脑。它要做的是一个窄而深的 AI 漫剧小说改编生产系统：输入小说，输出 AI 漫剧剧本和 AI 漫剧分镜，并让这两个产物在反复使用中越来越稳定。

## 一、三个参考来源各自提供什么

### 1. `0WcAssistant` 提供本地化判断

`0WcAssistant` 的价值不在于把个人驾驶舱搬过来，而在于它已经讨论过几条适合你的本地工作方式：

- 技能先作为本地蓝图沉淀，不急着安装成全局技能。
- 不要一开始凭空设计复杂角色，而是从真实动作里长出高频技能。
- 核心系统和业务模块要有边界，不能把领域规则随便吞进中枢。
- 文档、里程碑、数据协议要先落地，后续实现才不会跑偏。

迁移到本项目后，这些原则变成：

- 剧本和分镜技能先放在项目本地。
- 先做一个完整闭环，再拆成多个专业技能。
- 类型、题材、平台规则要放在可维护的参考规则中，而不是散落在聊天里。
- 每次生成都保存产物、反馈和版本，方便复盘。

### 2. `gstack` 提供流程化技能包思路

`gstack` 值得借鉴的是“把一个复杂过程拆成一组角色明确的技能”，并且每个技能都有明确的输入、输出和交接。

对本项目的映射：

| `gstack` 思路 | 本项目落点 |
| --- | --- |
| 需求澄清和规格编写 | 小说改编前先整理来源、目标、平台、篇幅、受众 |
| 计划评审 | 剧本生成前明确结构和风险 |
| 工程评审、设计评审 | 剧本评审、分镜评审 |
| 前一个技能产物交给下一个技能 | 剧本评审结果必须进入分镜生成 |
| 学习和上下文保存 | 保存用户偏好和项目经验 |

不借鉴：

- 软件开发发布流程
- 浏览器测试流程
- 手机端测试流程
- 持续集成、部署、安全命令集

### 3. `gbrain` 提供可进化记忆层

`gbrain` 值得借鉴的是“产物和经验可以被沉淀、检索、评测、优化”。

对本项目的映射：

| `gbrain` 思路 | 本项目落点 |
| --- | --- |
| 捕获和入库 | 保存小说输入、剧本、分镜、用户反馈 |
| 检索和综合 | 生成前检索同类型历史案例和规则 |
| 知识维护 | 定期检查技能规则是否过期、冲突、重复 |
| 技能优化 | 用评测样例判断技能修改是否真的变好 |
| 规则包 | 不同小说类型对应不同规则包 |

不借鉴：

- 完整个人知识脑
- 大规模知识图谱
- 多用户权限
- 远程工具服务
- 自动后台摄取所有个人资料

## 二、当前项目的核心架构判断

### 判断 1：先做一个总控技能，再拆专业技能

第一阶段不要马上拆成十几个技能。建议先做一个：

```text
skills/novel-to-script-storyboard/SKILL.md
```

它完成完整闭环：

```text
小说输入 -> 输入整理 -> 类型题材判断 -> 剧本生成 -> 剧本评审 -> 分镜生成 -> 分镜评审 -> 学习记录
```

当这个闭环跑过多个真实样例后，再拆分为：

- 小说输入整理
- 类型题材判断
- 剧本生成
- 剧本评审
- 分镜生成
- 分镜评审
- 技能学习

### 判断 2：类型和题材规则放参考文件，不塞进主技能

主技能只写流程和质量门槛。类型、题材、平台、格式规则放到参考文件：

```text
skills/novel-to-script-storyboard/
├── SKILL.md
├── references/
│   ├── genre-taxonomy.md
│   ├── script-format.md
│   ├── storyboard-format.md
│   ├── review-rubrics.md
│   ├── platform-rules.md
│   └── candidate-rules.md
```

这样后续新增“甜宠”“悬疑”“仙侠”“家庭伦理”“爽文逆袭”等类型时，不需要重写主技能。

### 判断 3：自我完善必须分层

不能每次用户不满意就自动改主技能。应该分三层：

1. **运行记录**：这次哪里不好，用户怎么改。
2. **候选规则**：重复出现的问题进入候选规则文件。
3. **正式规则**：通过人工审查和评测样例后，再进入主规则。

这既借鉴了 `gbrain` 的技能优化思路，也保留人工判断，避免一次坏样例污染系统。

### 判断 4：剧本和分镜之间必须有评审门

不要让弱剧本直接生成分镜。

分镜是剧本的视觉化延伸。如果剧本里的冲突、节奏、人物动机有问题，分镜只会把问题放大。

因此流程必须是：

```text
剧本草稿
-> 剧本评审
-> 剧本定稿
-> 分镜草稿
```

### 判断 5：产物协议比界面更早

当前阶段先定义文件结构，不急着做界面。

建议每次运行保存为：

```text
runs/<日期>-<项目名>/
├── source.md              原始小说
├── metadata.json          基础信息
├── synopsis.md            梗概
├── routing.json           类型和题材判断
├── script.draft.md        剧本草稿
├── script.review.md       剧本评审
├── script.final.md        剧本定稿
├── storyboard.draft.md    分镜草稿
├── storyboard.review.md   分镜评审
├── storyboard.final.md    分镜定稿
├── learnings.jsonl        学习记录
└── run-manifest.json      本次运行清单
```

有了这个协议，后续无论做命令行、网页、技能、还是自动评测，都有稳定输入输出。

## 三、推荐里程碑压缩版

### M0：方向冻结

产物：

- 项目说明
- 综合参考判断
- 技能系统设计
- 里程碑节点

当前已完成第一版。

### M1：总控技能原型

目标：能用一个技能完成从小说到剧本再到分镜。

产物：

- `skills/novel-to-script-storyboard/SKILL.md`
- 剧本格式参考
- 分镜格式参考
- 类型题材参考
- 评审规则参考

### M2：样例运行

目标：用 3 到 5 个不同类型小说片段跑通流程。

建议样例类型：

- 都市情感 / 甜宠
- 悬疑 / 惊悚
- 仙侠 / 玄幻
- 家庭伦理
- 爽文逆袭

### M3：评审和学习记录

目标：每次输出都能留下评审和学习记录。

产物：

- 剧本评审
- 分镜评审
- 学习记录
- 候选规则

### M4：评测样例驱动改进

目标：开始判断技能修改是否真的更好。

产物：

- 类型判断评测样例
- 剧本生成评测样例
- 分镜生成评测样例

### M5：拆分专业技能

目标：从总控技能拆成多个职责清晰的专业技能。

拆分时机：

- 总控技能已跑过足够真实样例。
- 输出格式稳定。
- 常见问题已经能归类。
- 不同类型小说确实需要不同规则。

## 四、当前最合适的下一步

下一步不要先做页面，也不要先做完整知识库。

最合适的是：

```text
M1.1 创建 skills/novel-to-script-storyboard/SKILL.md
```

并同步创建最小参考文件：

```text
skills/novel-to-script-storyboard/references/
├── genre-taxonomy.md
├── script-format.md
├── storyboard-format.md
├── review-rubrics.md
└── candidate-rules.md
```

第一版只要能稳定回答一个问题：

> 给我一段小说，我能否得到结构稳定、可评审、可继续迭代的剧本和分镜？

能做到这一点，再谈自我完善和拆技能。
