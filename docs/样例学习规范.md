# 样例学习规范

本文档说明已有 AI 漫剧剧本和分镜应该如何放入项目，以及系统应该从中学习什么。

## 一、为什么要学习已有样例

已有 AI 漫剧剧本和分镜体现了真实工作习惯，比凭空设计规则更可靠。

它们可以帮助系统学习：

- 剧本格式
- 分镜格式
- 场景拆分方式
- 镜头密度
- 旁白和对白比例
- AI 漫剧画面生成友好度
- AI 漫剧字幕、音效和节奏习惯
- 不同题材的画面风格
- 用户喜欢和不喜欢的写法
- 哪些问题经常需要人工修改

## 二、样例目录

本项目支持两种样例投放方式。

重要原则：使用人员只负责提供材料，不负责后续整理。

使用人员可以把文件直接放进 `samples/` 或 `samples/_inbox/`，也可以在对话窗口里直接发送小说、剧本、分镜或混合材料。

使用人员不需要手动改文件名、判断剧本和分镜关系、建立规范目录、编写 `notes.md` 或提炼规则。这些工作都由系统完成。

如果系统无法根据文件名、集数、正文人物、地点和剧情判断文件之间的对应关系，才向使用人员提问确认。提问只用于确认关键关系，不把整理工作交回给使用人员。

### 方式一：一股脑原始投放

这是默认推荐方式。你不需要先整理，直接把以前做过的剧本、分镜、聊天记录、导出文档、片段文件都放进：

```text
samples/_inbox/
```

如果你已经直接放在 `samples/` 根目录，也没关系。系统也会把 `samples/` 根目录下的散落文件视为待分拣原始材料。

允许的状态：

- 文件名很乱。
- 剧本和分镜混在一个文件里。
- 一个文件里有多个项目。
- 只有剧本，没有分镜。
- 只有分镜，没有剧本。
- 有重复版本。
- 有草稿、定稿、废稿混在一起。
- 没有说明来源。

系统的任务不是要求你先整理，而是先做“样例分拣”：

```text
samples/_inbox/ 或 samples/ 根目录散落文件
-> 识别材料类型
-> 判断可能属于哪个项目
-> 拆分剧本 / 分镜 / 说明
-> 标记草稿 / 定稿 / 不确定
-> 生成待确认清单
-> 再整理到 samples/剧目/<编号>-<剧名>/episodes/<集数>-<集名>/
```

### 方式二：按项目整理投放

如果后续有时间，也可以把样例整理成。注意：这不是对使用人员的要求，而是系统整理后的目标结构：

```text
samples/
├── README.md
├── _inbox/
├── _reports/
└── 剧目/
    └── 001-剧名/
        ├── 剧目说明.md
        ├── source/            可选，小说或原始材料
        ├── scripts/           可选，总剧本或多集剧本
        └── episodes/
            └── 001-第一集/
                ├── script.input.md
                ├── script.final.md
                ├── storyboard.input.md
                ├── storyboard.final.md
                └── notes.md
```

核心原则：先按“一部剧 / 一个项目”归拢，再按集数整理。不要把同一部剧的第 2 集、第 3 集分镜拆成互不相关的平铺样例。

如果某个剧目没有小说原文，也可以只放剧本和分镜。如果某一集只有剧本或只有分镜，也可以放入对应集目录，但必须在 `notes.md` 说明缺口。

## 三、原始投放后的整理结果

`sample-ingest` 处理 `_inbox` 或 `samples/` 根目录散落文件后，应该输出一份分拣报告，而不是假装所有材料都已经干净。

推荐输出：

```text
samples/_reports/
├── inbox-scan-日期.md
└── unresolved-日期.md
```

分拣报告要回答：

- 找到了哪些文件。
- 每个文件大概是什么：剧本、分镜、混合材料、未知。
- 哪些文件可能属于同一个项目。
- 哪些文件之间存在明确配对关系，例如同一故事的剧本和分镜。
- 哪些文件虽然不是一一配对，但从文件名能看出属于同一故事或同一系列。
- 哪些像草稿，哪些像定稿。
- 哪些可以直接学习。
- 哪些需要你确认。
- 哪些暂时不要学习。

## 四、样例关联关系

样例学习时，必须考虑文件之间的关联性。

第一层先看文件名。很多历史样例不一定有说明文档，但文件名本身已经透露了项目名、集数和材料类型。

常见关联关系：

- 小说原文 -> 剧本
- 剧本 -> 分镜
- 小说原文或总剧本 -> 多集分镜
- 多集剧本 -> 其中若干集分镜
- 剧本草稿 -> 剧本定稿
- 分镜草稿 -> 分镜定稿
- 同一故事的不同集
- 同一故事的不同版本

文件名判断时，先抽取三类信息：

- 共享标题词：如“新房”“八零”“欧皇”“少奶奶”“第十七集”。
- 集数词：如“第一集”“第二集”“第3集”“第十七集”。
- 材料类型词：如“小说”“剧本”“改写稿”“分镜”“完本”。

只要共享标题词和材料类型词能对应上，就应先放进同一关联组，再通过正文内容二次确认。

例如一：

```text
samples/新房剧本.docx
samples/新房分镜.docx
```

这两个文件应识别为同一个故事的“剧本 -> 分镜”配对样例，而不是两个互不相关的文件。

例如二：

```text
samples/八零杂货铺.docx
samples/八零杂货铺_AI漫剧前10集剧本改写稿.md
samples/八零分镜第二集.docx
samples/八零分镜第3集分镜.docx
```

这些文件应先识别为“八零杂货铺”同一故事组。即使还没读取正文，也不能孤立学习其中任何一个文件。

如果一个总剧本只对应目前已提供的第 2 集、第 3 集分镜，这是正常的一对多关系，不要因为缺少第 1 集或其他集分镜就判断材料异常。系统只整理已提供的分集分镜，并记录“当前已收到哪些集”。

例如三：

```text
samples/【完本】《修仙：欧皇盲盒带我躺成天道之主》(1).docx
samples/欧皇分镜第一集.docx
samples/欧皇第二集分镜.docx
samples/欧皇第三集分镜.docx
```

这些文件应先识别为“欧皇盲盒 / 修仙”同一系列组。由于目前缺少明确剧本文件，先作为“小说或原始材料 -> 多集分镜”的关联组处理。

关联样例的学习价值更高，因为它可以直接分析：

- 剧本内容如何被拆成分镜。
- 哪些剧本信息进入了画面。
- 哪些对白保留到了分镜。
- 每场剧本大概拆成多少镜头。
- 分镜如何补充景别、运镜、音效和时长。

## 五、部分对应样例

历史样例不一定都是完整的一一对应。文件名和集数能确认属于同一故事组，但正文可能只覆盖部分场次，或者分镜来自后续改写版本。

遇到这种情况，不要把材料退回给使用人员整理，也不要强行当作完整配对。系统应标为“部分对应样例”：

- 仍然建立规范样例目录。
- 在 `notes.md` 写明已覆盖场次、缺失场次和可能改写部分。
- 只学习明确覆盖的镜头拆分、节奏、物品画面、景别、运镜和音效。
- 不把缺失场次总结成“可以删除”的规则。
- 如果缺口影响关键判断，再向使用人员提出简短确认问题。

例如八零杂货铺：

- 八零杂货铺有小说、剧本和分镜，但当前只提供了第 2 集和第 3 集分镜。这属于“总材料 -> 部分分集分镜”的正常状态，不是异常缺漏。
- 第二集分镜从柳河镇主街开始，未覆盖第 2 集剧本开头林家堂屋逼婚场，因此这一集内部只能作为部分对应样例。
- 第三集分镜覆盖摆摊成交和回家亮钱，但未覆盖该集糖纸暴露秘密场，因此不能据此学习“尾钩可以删除”。

## 六、每组样例的说明文件

整理后的每组样例必须有 `notes.md`。默认由系统自动生成待确认版本，不要求使用人员手写：

```markdown
# 样例说明

## 基础信息

- 类型：
- 题材：
- 目标平台：
- 目标时长：
- 是否满意：

## 值得学习的地方

- 剧本结构：
- 分镜格式：
- 镜头节奏：
- 画面风格：

## 不希望学习的地方

- 问题 1：
- 问题 2：

## 用户修改记录

- 修改前：
- 修改后：
- 修改原因：
```

## 七、学习时提炼什么

系统从样例中提炼四类内容。

### 1. 格式规则

例如：

- 剧本是否按场次编号。
- 是否区分旁白、对白、动作。
- 分镜表有哪些字段。
- 镜头编号格式是什么。

### 2. 风格规则

例如：

- 悬疑类开头是否需要先给异常事件。
- 甜宠类对白是否更口语化。
- 仙侠类是否需要更多环境和气氛镜头。
- 家庭伦理类是否强调表情和沉默反应。

### 3. 质量规则

例如：

- 哪些剧本不能直接进入分镜。
- 哪些分镜描述太空。
- 哪些镜头不可拍。
- 哪些转场缺失。

### 4. 用户偏好

例如：

- 喜欢短镜头还是长镜头。
- 喜欢强旁白还是弱旁白。
- 喜欢密集分镜还是简洁分镜。
- 喜欢保守改编还是强改编。

### 5. 文件关联规则

例如：

- `新房剧本.docx` 与 `新房分镜.docx` 是同一故事配对。
- `第十七集 - 剧本.doc` 与 `第十七集 - 分镜.doc` 是同一集配对。
- `八零杂货铺_AI漫剧前10集剧本改写稿.md` 与 `八零分镜第二集.docx`、`八零分镜第3集分镜.docx` 属于同一故事的一对多关系：总剧本覆盖多集，目前只收到第 2 集和第 3 集分镜。
- `欧皇分镜第一集.docx`、`欧皇第二集分镜.docx`、`欧皇第三集分镜.docx` 与欧皇小说属于同一系列。
- `救命！司爷他快镇不住少奶奶了.txt` 与 `少奶奶分镜.doc` 从“少奶奶”关键词可判断为同一故事组。

## 八、学习结果怎么进入技能

学习结果不要直接写入正式技能。

推荐流程：

```text
样例
-> 提炼观察
-> 候选规则
-> 人工确认
-> 正式规则
-> 后续评测
```

这样既能让系统吸收经验，又不会因为某个样例质量一般而污染整体规则。

## 九、当前建议

当前项目目录里已经有一批实际样例，且多数可以从文件名看出关联关系。

你后续仍然可以继续把材料直接放入：

```text
samples/_inbox/
```

已经放在 `samples/` 根目录的材料也会直接处理，不需要你再手动搬。

系统会先做分拣。整理后才会形成类似下面的规范目录：

```text
samples/剧目/001-你的剧名/
├── 剧目说明.md
└── episodes/
    └── 001-第一集/
        ├── script.input.md
        ├── script.final.md
        ├── storyboard.final.md
        └── notes.md
```

放入后，系统优先学习：

1. 剧本格式。
2. 分镜字段。
3. 镜头密度。
4. 用户修改偏好。
5. 可以进入候选规则的稳定经验。
