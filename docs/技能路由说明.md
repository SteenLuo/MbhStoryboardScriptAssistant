# 技能路由说明

本文说明使用者给出不同材料时，系统应该调用哪类技能。

本项目只服务两个核心产物：AI 漫剧剧本、AI 漫剧分镜。其他能力都是为了这两个产物稳定生成、评审和学习。

## 一、总入口

总控技能：

- `skills/00-orchestrator/mbh-workflow`

作用：

- 判断输入材料类型。
- 选择三条主链路之一。
- 创建或使用运行目录。
- 在结束前判断是否需要学习记录。

## 二、路由表

| 用户输入 | 首选技能 | 后续技能 | 说明 |
| --- | --- | --- | --- |
| 小说原文 / 章节 | `novel-intake` | `script-generate`、`script-review-rewrite`、`storyboard-generate` | 先做正向分析和负面分析 |
| 故事梗概 / 人设 / 世界观 | `novel-intake` | `script-generate` | 先整理成可改编信息 |
| 已有剧本但质量未知 | `script-review-rewrite` | `storyboard-generate` | 先判断能否进入分镜 |
| 已认可剧本 | `storyboard-generate` | 分镜评审和学习记录 | 可跳过剧本改写 |
| 已有分镜 | `storyboard-generate` 或 `sample-ingest` | 分镜评审、样例学习 | 看用户是要评审还是学习 |
| 一堆历史样例 | `sample-ingest` | `skill-evolution` | 先分拣，不要求用户整理 |
| 用户修改意见 | `skill-evolution` | `New-ConversationLearningRecord` | 判断是否长期偏好 |
| 用户说质量下降 | `skill-evolution` | `New-QualityRegressionRecord` | 先止损，再复核 |

## 三、触发样例

### `novel-intake`

触发话术：

- “这是小说第一章，帮我整理一下。”
- “先根据小说形成大纲和每集爽点。”
- “检查这段剧情有没有逻辑问题。”

输出：

- 小说大纲。
- 每集大纲。
- 每集 2 到 4 个爽点或高潮点。
- 单集和整体负面逻辑分析。

### `script-generate`

触发话术：

- “根据这段小说生成 AI 漫剧剧本。”
- “先写成一集剧本。”
- “按爽点和冲突改编成剧本。”

输出：

- 分场剧本。
- 旁白、对白、动作。
- 改编说明和风险。

### `script-review-rewrite`

触发话术：

- “评审这个剧本能不能进入分镜。”
- “这个剧本先改一下再分镜。”
- “判断它是直接过、轻改还是重写。”

输出：

- 通过 / 轻改 / 重写。
- 问题清单。
- 修订剧本。

### `storyboard-generate`

触发话术：

- “根据这个剧本生成分镜。”
- “这个剧本已经认可，直接分镜。”
- “分镜不通过就小修。”

输出：

- 标准分镜。
- 分镜评审。
- 必要时的小修版分镜。

### `sample-ingest`

触发话术：

- “这些样例学习一下。”
- “我把剧本和分镜都扔进 samples 了。”
- “这些文件名能看出对应关系。”

输出：

- 分拣报告。
- 关联关系。
- 候选规则。
- 待确认问题。

### `skill-evolution`

触发话术：

- “把这个规则补进技能里。”
- “这个问题出现好几次了。”
- “学习后质量下降了。”

输出：

- 技能进化草案。
- 候选规则或已确认规则建议。
- 降质和回退建议。

## 四、交接产物

| 上游 | 下游 | 交接产物 |
| --- | --- | --- |
| `novel-intake` | `script-generate` | 输入分析、小说大纲、每集大纲、爽点和负面分析 |
| `script-generate` | `script-review-rewrite` | 剧本草稿 |
| `script-review-rewrite` | `storyboard-generate` | 通过评审或修订后的剧本 |
| `storyboard-generate` | `skill-evolution` | 分镜、分镜评审、覆盖核对、用户反馈 |
| `sample-ingest` | `skill-evolution` | 分拣报告、候选规则、对齐复核 |
| 对话反馈 | `skill-evolution` | 对话学习记录、降质记录 |

## 五、路由原则

- 输入不明时，先整理，不跳步。
- 剧本未认可时，先评审，不直接分镜。
- 分镜不通过时，必须选择去向：小修、重新生成、返回剧本评审。
- 样例关系不确定时，优先按剧目归拢；实在无法判断才询问使用人员。
- 对话里的长期偏好要记录，普通问答静默跳过。

## 六、网页按钮执行规则

网页中的“生成剧本”和“生成分镜”不是普通提示词按钮，而是标准流程按钮。

### 生成剧本

用户点击“生成剧本”时，后端必须按以下顺序执行：

```text
input.md
-> input-analysis.md
-> generated-script.md
-> script-review.md
```

也就是说，即使用户直接上传小说原文并点击“生成剧本”，也不能跳过 `novel-intake`。生成剧本后要同步生成剧本评审，作为后续进入分镜的前置门槛。

### 生成分镜

用户点击“生成分镜”时，后端必须先检查是否存在剧本和剧本评审：

```text
input.md
-> input-analysis.md
-> generated-script.md 或 用户提供剧本
-> script-review.md
-> generated-storyboard.md
-> storyboard-review.md
```

如果没有 `script-review.md`，必须先生成剧本评审；如果评审没有明确通过或轻改后通过，流程停在剧本评审，不生成分镜。

### 产物显示

流程图只展示运行目录中的阶段产物，不把普通聊天内容假装成已完成节点。阶段是否完成以对应文件是否存在为准。
