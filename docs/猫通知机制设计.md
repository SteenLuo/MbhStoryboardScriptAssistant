# 猫通知机制设计

日期：2026-07-01

本文定义左下角猫通知机制。通知不是学习记录本身，而是“需要用户看一眼”的入口。第一版用于学习失败，后续可扩展到吃饭、下班等关怀提醒。

## 一、定位

1. 通知只提示需要处理或查看的事件，不保存完整业务证据。
2. 学习失败的证据仍以学习事件账本为准。
3. 通知应持久未读，刷新或重启后仍可看到。
4. 通知不能阻塞用户创作。

## 二、交互

通知窗口从左下角猫附近弹出，视觉上指向猫。

一次显示一条通知。多条通知显示序号，例如 `1/5`、`2/5`。处理完最后一条后，通知窗口关闭。

按钮：

- 确定：执行主动作，例如跳转到学习资料库对应失败详情。
- 取消：仅处理当前通知，不删除业务记录。

第一版通知排序：

1. 学习失败优先。
2. 同类型按创建时间顺序。

## 三、数据

建议文件：`app/data/notifications.json`

字段：

- `id`
- `type`
- `sourceId`
- `sourceType`
- `title`
- `summary`
- `target`
- `status`
- `createdAt`
- `handledAt`

状态：

- `unread`
- `handled`
- `withdrawn`

## 四、学习失败接入

学习事件最终转为“失败”时，生成通知：

- 标题：学习失败
- 摘要：例如“分镜台词长度规则未能发布”
- 目标：学习资料库 -> 学习记录 -> 对应事件详情

不在每次重试失败时弹通知。只有最终失败才通知。

如果失败事件随后被新事件覆盖，未读通知应撤销或转为已处理，避免提示用户处理已经失效的旧失败。

## 五、与猫关怀提醒的关系

当前已有轻提醒适合短句自动消失。新通知机制适合有动作、有排队、有持久状态的事件。

后续关怀提醒可以复用通知队列，但类型和优先级应与学习失败区分：

- `learning`
- `system`
- `care`

学习失败不应被关怀提醒覆盖或挤掉。
