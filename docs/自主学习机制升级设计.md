# 自主学习机制升级设计

日期：2026-07-01

本文是 1.0 版本自主学习机制的主设计文档。它替代旧 M5 中“候选规则必须人工确认后才进入正式规则”的单一路径，但不删除旧学习资产。旧资产作为历史证据迁移，新机制以“当前规则层自动发布”为核心。

## 一、核心原则

1. 学习机制是系统核心能力，不是流程图或工作台的附属功能。
2. 用户正常创作时，生成链路只读取“当前规则层”和必要技能，不直接扫描学习事件账本。
3. 学习事件账本只做原始证据留痕，追加记录，不物理覆盖，不直接参与实时生成。
4. 明确规则可以自动生效，但必须通过发布校验。发布失败不能伪装成已学会。
5. 取消“兜底”表述，统一使用：
   - 发布保护：坏规则不覆盖 last-good 当前规则层。
   - 故障隔离：学习失败不影响对话、画布、剧本和分镜生成。
   - 当前对话上下文执行：用户刚提出的规则，本轮对话先按上下文执行。
6. Skill 是能力入口和方法论，当前规则层是 skill 调用前加载的生效规则，小技能只在方法论稳定时创建。

## 二、学习入口

第一版支持四类入口：

1. 对话明确规则：例如“以后分镜台词每句 20 字以内”。
2. 对话反复纠错：用户多次要求同类格式、风格或限制。
3. 样例投喂：用户提供小说、剧本、分镜或修改稿，让系统学习写法。
4. 画布归档并学习：最终版画布冻结后，后台基于小说、剧本、分镜和修改链路做慢速学习。

快速通道用于明确规则。对话回复口径为：“这次先按你的要求执行，长期记忆正在处理。”不能在发布成功前说“已记住”。

慢速通道用于样例投喂和画布归档。慢速学习不能阻塞用户创作。

## 三、数据分层

### 学习事件账本

建议文件：`learning/events.jsonl`

每条事件至少包含：

- `eventId`
- `topicKey`
- `sourceType`
- `projectId`
- `canvasId`
- `conversationId`
- `status`
- `summary`
- `rawTrigger`
- `createdAt`
- `updatedAt`
- `tokenUsage`
- `error`
- `coveredByEventId`

用户侧状态只显示：

- 处理中
- 已生效
- 已被覆盖
- 失败

不显示“已忽略”等未确认业务状态。

### 当前规则层

建议文件：`learning/current-ruleset.json`

它是生成链路读取的长期生效规则，不是 skill 本身。每条规则至少包含：

- `ruleId`
- `topicKey`
- `capability`
- `content`
- `priority`
- `sourceEventIds`
- `status`
- `createdAt`
- `updatedAt`

当前规则层规则状态：

- `active`：已启用 / 已生效。会在命中对应 skill 时加载进模型上下文。
- `disabled`：已停用。用户手动停用后仍保留在当前规则页，但生成链路不加载、不调用。
- `covered`：已被覆盖。被同主题后续规则替代，只读展示，不允许直接一键启用；如需恢复旧方向，应通过正常对话重新学习生成新规则。

当前规则层发布前必须通过：

1. JSON 结构校验。
2. 必填字段校验。
3. 加载校验。
4. 同主题冲突校验。

通过后发布为 last-good。失败时继续使用上一版 last-good。

### 技能库索引

建议文件：`learning/skill-index.json`

技能库展示基础技能和自动创建的小技能。小技能采用严格闸门：

- 单条规则不创建小技能。
- 低置信样例不创建小技能。
- 单项目特殊偏好不创建小技能。
- 只有完整方法论或多条稳定规则支持时，才创建并生效。

用户侧称为“技能库”，内部可称 derived skill。

## 四、队列、覆盖与重试

学习后台队列按主题并行。不同 `topicKey` 可以并行，同一 `topicKey` 内最新任务独占发布权。

同主题新事件覆盖旧事件时：

1. 旧 pending、retrying、processing 任务立即失去发布权。
2. 可取消则取消；无法中断的模型请求返回后直接丢弃。
3. 旧事件保留账本证据，状态显示“已被覆盖”。
4. 新任务可以把旧事件作为同主题证据，但发布权只属于最新任务。

覆盖判定采用规则加模型：

1. 程序先按能力域、字段、关键词归 `topicKey`。
2. 同主题或相近主题再由模型判断关系：
   - supplements
   - narrows
   - replaces
   - conflicts
   - unrelated
3. 只有 `narrows`、`replaces`、`conflicts` 触发覆盖。
4. 判断不准时不覆盖，只归入同主题待归纳。

失败重试采用分类策略：

- 网络超时、模型接口临时失败、文件占用、预算暂时不足、本地服务暂不可用：最多重试 3 次，建议间隔 1 分钟、5 分钟、15 分钟。
- 字段缺失、规则卡不可解析、JSON 结构错误、冲突无法判断、发布后加载失败：直接失败，不自动重试。

同主题新事件覆盖旧事件时，旧事件的所有重试立即终止。

## 五、学习资料库

左下角新增“学习资料库”入口。页面只读，不做规则编辑器。

三类 Tab：

1. 学习记录：看发生过什么、状态、来源、token、失败原因和覆盖关系。
2. 当前规则：看现在长期规则及启用状态；未覆盖规则可停用或启用，已覆盖规则只读。
3. 技能库：看基础技能和已创建的小技能。

失败详情需要支持复制完整 JSON，内容包括：

- eventId
- 阶段
- 错误码
- message
- stack 摘要
- retryCount
- nextRetryAt
- tokenUsage
- source paths
- lastGoodVersion
- 是否影响创作
- 是否已被覆盖

## 六、Token 统计

入账不耗 token。明确规则可使用短 prompt 或模型归纳，优先保证效果。

学习 token 与创作 token 分开统计。第一版只统计不限制，不设置硬预算，不让学习因为预算策略看起来失灵。

## 七、旧资产迁移

旧资产迁移采用“只读索引 + 已确认种子”：

1. `learning/conversation-records`
2. `learning/candidate-rules`
3. `learning/regression-reports`
4. `learning/snapshots`
5. `learning/skill-evolution-reports`

这些进入学习记录只读视图，不直接生效。

`learning/accepted-rules` 可种子化当前规则层，但仍必须经过结构、加载、冲突校验。

## 八、生成链路关系

生成流程：

任务路由 -> 基础 skill -> 当前规则层相关规则 -> 必要小技能 -> 生成 -> 硬规则检查。

硬规则检查不属于学习成功，也不是发布保护。它负责检查模型输出是否真的满足可程序判定的交付规则。

分镜台词每句 20 字以内这类规则，必须同时进入当前规则层和分镜生成后检查。
